# RepoSync 系统总览

## 1. 产品定位
RepoSync 首版是一个单机部署的 Git 镜像同步管理台，面向单管理员使用。系统负责把一个或多个源仓库镜像同步到目标平台，并在需要时自动创建目标仓库。系统同时支持递归处理子模块仓库，并通过本地裸仓库缓存减少重复同步耗时。

## 2. 总体架构
建议采用单进程分层架构：

```text
+-----------------------------+
|         Vue 管理台          |
+-------------+---------------+
              |
              v
+-----------------------------+
|        HTTP API 层          |
+-------------+---------------+
              |
              v
+-----------------------------+
|       应用服务层            |
|  Task / Credential / Run    |
+------+------+---------------+
       |      |
       |      +-------------------+
       |                          |
       v                          v
+-------------+          +-----------------+
| 调度器      |          | 同步执行引擎     |
| Cron/Webhook|          | 执行树/失败链路  |
+-------------+          +-----------------+
                                  |
          +-----------------------+------------------------+
          |                       |                        |
          v                       v                        v
+-----------------+     +------------------+     +------------------+
| Git 适配层      |     | SCM Provider     |     | 缓存管理器       |
| clone/fetch/push|     | GitHub/Gogs API  |     | bare mirror lock |
+-----------------+     +------------------+     +------------------+
          |
          v
+-----------------------------+
| SQLite + 本地缓存目录       |
+-----------------------------+
```

## 3. 模块划分

### 3.1 后端模块
- `cmd/server`
  - 服务启动入口
  - 读取配置、初始化依赖、启动 HTTP 服务
- `internal/app`
  - 应用初始化
  - 路由注册
  - 配置加载
- `internal/domain`
  - 领域实体与枚举
  - 输入输出模型
- `internal/store`
  - SQLite 持久化
  - 任务、凭证、执行记录、缓存元数据的读写
- `internal/service`
  - 任务 CRUD
  - 凭证 CRUD
  - 手动执行、Webhook 执行、定时执行编排
- `internal/git`
  - 系统 `git` 命令封装
  - 本地缓存 mirror 初始化与 fetch
  - push --mirror
  - 子模块解析
- `internal/scm`
  - GitHub / Gogs 平台适配
  - 目标仓库存在性检查
  - 缺失仓库自动创建
- `internal/security`
  - 本地密文存储
  - 敏感信息脱敏

### 3.2 前端模块
- `src/views`
  - 任务列表页
  - 任务编辑页
  - 执行详情页
  - 缓存页
  - 凭证页
- `src/components`
  - 任务表单
  - 执行树
  - 凭证表单
  - 缓存状态卡片
- `src/api`
  - 对接后端 REST API
- `src/router`
  - 路由与页面组织

## 4. 核心执行链路

### 4.1 手动/定时/Webhook 统一执行流程
1. 根据任务 ID 读取 `SyncTask`
2. 校验任务是否启用
3. 生成一条 `SyncExecution`
4. 加载源和目标凭证
5. 校验目标仓库是否存在
6. 若不存在则调用 GitHub/Gogs API 自动建仓
7. 获取源仓库本地缓存锁
8. 初始化或更新本地 bare mirror 缓存
9. 解析主仓库子模块信息
10. 递归处理所有子模块
11. 子模块全部完成后，对主仓库执行镜像推送
12. 汇总状态、生成执行树、写回执行结果

### 4.1.1 镜像同步语义
- 主仓库同步必须采用完整镜像语义，而不是仅同步默认分支
- 需要镜像源仓库中的所有分支引用
- 需要镜像源仓库中的所有标签引用
- 如果源仓库存在其他 Git refs，首版默认随 `--mirror` 一并推送
- 目标端应尽量保持与源端 refs 一致
- 首版不提供“只同步指定分支”的裁剪模式

### 4.2 子模块递归策略
- 通过 `.gitmodules` 解析子模块路径和源仓库 URL
- 通过 `git ls-tree` 获取子模块引用 commit
- 使用深度优先遍历构造执行树
- 使用 `visited` 集合检测循环依赖和重复仓库
- 任一子模块失败，主任务整体失败
- 在主仓库推送到目标端前，先根据子模块目标映射改写 `.gitmodules`
- 改写后的主仓库分支会覆盖推送到目标端，确保目标仓库 `clone --recurse-submodules` 默认指向镜像后的子模块地址
- 完整镜像 refs 仍会先推送一次，以保留标签和其他 refs；随后仅对分支 refs 覆盖推送改写后的提交

### 4.3 目标仓库映射规则
- 主仓库使用任务配置中的 `targetRepoUrl`
- 子模块目标仓库按“主仓库名 + 子模块路径扁平化”规则派生
- 示例
  - 主仓库：`team/main.git`
  - 子模块路径：`libs/core/http`
  - 子模块目标：`team/main-libs-core-http.git`

首版不提供自定义映射表达式，管理台只做映射预览。

### 4.4 目标主仓库可用性保证
- 系统不仅同步子模块仓库本身，还会同步改写主仓库中的 `.gitmodules`
- 改写规则使用子模块路径对应的目标镜像仓库地址
- 这样目标端用户直接执行 `git clone --recurse-submodules` 时，子模块默认拉取的是镜像后的目标仓库，而不是源仓库

## 5. 自动建仓设计

### 5.1 支持范围
- 主仓库目标自动建仓
- 子模块目标自动建仓
- 平台支持：GitHub、Gogs

### 5.2 创建时机
- 同步过程中发现目标仓库不存在时立即创建
- 创建成功后继续执行镜像推送
- 如果已存在则直接跳过

### 5.3 建仓参数
- 组织或用户命名空间
- 仓库可见性：`private` / `public`
- 仓库描述模板
- 平台 API Base URL

### 5.4 风险控制
- 建仓请求要求幂等
- 对命名冲突、权限不足、组织不存在给出明确错误
- 执行详情中记录“是否自动建仓”和“建仓耗时”

## 6. 缓存设计

### 6.1 缓存键
缓存键由以下信息组成：
- 规范化后的源仓库 URL
- 认证上下文摘要

### 6.2 缓存形态
- 每个源仓库使用一个本地 bare mirror 仓库
- 主仓库和子模块分别命中各自缓存
- 缓存长期保留，默认不自动删除

### 6.3 缓存流程
1. 根据缓存键定位缓存目录
2. 若缓存不存在则 `git clone --mirror`
3. 若缓存已存在则执行完整 refs 增量抓取，确保所有分支和标签都被更新
4. 记录最近 fetch 时间、最近使用时间、大小、命中次数

### 6.4 并发控制
- 同一任务仅允许一个执行实例
- 同一缓存仅允许一个 fetch/push 更新流程
- 需要同时使用进程内 keyed mutex 和文件锁时，优先保证实现简单可维护

## 7. 安全设计
- 凭证不明文返回给前端
- 数据库存储使用对称加密保存敏感信息
- 日志中统一脱敏
- 默认关闭任何超出镜像推送必要范围的危险操作
- 首版不实现多用户和 RBAC

## 8. 部署建议
- 单机部署
- 后端默认监听 `:8080`
- 前端构建产物可由后端静态托管
- SQLite、缓存目录、日志目录统一放入 `data/`
