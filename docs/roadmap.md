# RepoSync 开发里程碑

## Phase 0: 工程基线
目标：把项目从空仓库推进到“可启动、可扩展”的基础状态。

交付物：
- 后端目录结构、配置加载、HTTP 服务入口
- 前端 Vue + Vite 工程骨架
- 基础 README 与开发文档

完成标准：
- 后端可启动并返回健康响应
- 前端可本地启动并显示基础管理台页面

## Phase 1: 基础数据能力
目标：先把管理台依赖的数据模型与 CRUD 打通。

交付物：
- SQLite 初始化与迁移脚本
- `SyncTask`、`Credential`、`RepoCache`、`SyncExecution`、`SyncExecutionNode` 表结构
- 任务 CRUD API
- 凭证 CRUD API
- 缓存列表 API

完成标准：
- 管理台可以新增、编辑、删除任务和凭证
- 凭证敏感信息不会明文回显

## Phase 2: 执行引擎 MVP
目标：打通“手动触发一次主仓库镜像同步”。

交付物：
- 手动触发接口
- 执行记录表写入
- 本地 bare mirror 缓存创建与复用
- 主仓库镜像推送，覆盖所有分支和标签
- 执行状态与摘要日志

完成标准：
- 首次同步可成功创建缓存并完成推送
- 第二次同步能命中缓存

当前状态：
- 已完成真实 Git mirror 流程
- 已通过本地集成测试验证“镜像所有分支、标签和其他 refs 的 mirror 语义”
- 自动建仓尚未接入，目标仓库当前仍需预先存在

## Phase 3: 自动建仓
目标：把目标仓库缺失时的自动创建能力接入主流程。

交付物：
- GitHub Provider
- Gogs Provider
- 目标仓库存在性检查
- 缺失仓库自动创建
- 建仓结果记录到执行节点

完成标准：
- 目标仓库不存在时，可自动建仓并继续同步
- 已存在仓库不会重复创建

当前状态：
- 尚未开始实现 Provider API 调用
- 已完成 Provider 抽象与 GitHub/Gogs 基础实现
- 已完成本地 bare 目标仓库自动初始化
- 已接入主仓库执行前的目标仓库存在性检查与缺失即创建
- 子模块递归建仓将在 Phase 4 一并接入

## Phase 4: 递归子模块同步
目标：支持主仓库携带多层子模块时的递归同步。

交付物：
- `.gitmodules` 解析
- 子模块引用 commit 获取
- 递归执行树
- 循环依赖检测
- 子模块目标仓库映射

完成标准：
- 多层子模块可递归同步
- 子模块失败时，主任务整体失败且错误链路可追踪

当前状态：
- 已完成 `.gitmodules` 解析
- 已完成子模块引用 commit 获取
- 已完成递归执行树与循环依赖检测
- 已完成按路径派生的子模块目标仓库映射
- 已完成主仓库 `.gitmodules` 重写，使目标主仓库默认引用镜像后的子模块地址
- 已完成改写提交稳定化，避免同一源分支重复同步时持续产生新的目标分支 SHA
- 已通过本地集成测试验证主仓库与子模块目标仓库的缺失即创建
- 当前 gitlink 仍保留源子模块提交，多层递归场景下尚未改写为“同步后的子模块提交”
- 前端树形可视化仍可继续增强

## Phase 5: 调度与 Webhook
目标：补齐非手动触发方式。

交付物：
- 定时任务调度
- GitHub Webhook
- Gogs Webhook
- 基础幂等和并发控制

完成标准：
- 定时和 Webhook 都能触发既有任务
- 同一任务不会并发执行多次

当前状态：
- 已完成基于 cron 表达式的定时调度
- 已完成任务级调度注册和反注册
- 已完成 GitHub/Gogs Webhook 签名校验
- 已完成 Webhook push 事件过滤和按任务目标分支的 ref 过滤
- Webhook 被忽略原因的前端展示仍可继续增强

## Phase 6: 管理台完善
目标：把所有后端能力沉淀到页面上。

交付物：
- 任务列表页
- 任务编辑页
- 执行详情树
- 缓存管理页
- 凭证页

完成标准：
- 管理员可完全在页面中完成首版主要操作

当前状态：
- 已完成任务状态、调度摘要、Webhook 摘要展示
- 已完成执行详情中的递归树视图
- 已完成任务列表中的调度、Webhook、分支过滤状态标签
- 已完成执行详情中的可折叠执行树、失败节点过滤和触发状态面板
- 已完成任务编辑区中的触发配置预览和 Webhook 路径预览
- 已完成执行树节点选中高亮与独立节点详情面板
- 缓存、凭证、执行历史页面已可用
- 更细的 Webhook 反馈视图、节点高亮和专门的调度状态页面仍可继续增强

## 测试优先级
- P0：任务 CRUD、凭证 CRUD、缓存查询、手动触发
- P1：自动建仓成功/失败、缓存命中/损坏恢复
- P2：多层子模块、循环依赖、Webhook、定时触发

## 推荐开发顺序
1. 先完成 `Phase 1 + Phase 2`
2. 再做 `Phase 3`
3. 然后接 `Phase 4`
4. 最后补 `Phase 5 + Phase 6`
