# RepoSync 开发路线图

## Phase 0: 工程基线

目标：把项目从空仓库推进到“可启动、可扩展”的基础状态。

交付物：
- 后端目录结构、配置加载、HTTP 服务入口
- 前端 Vue + Vite 工程骨架
- README 与开发文档

状态：已完成

## Phase 1: 基础数据能力

目标：先把管理台依赖的数据模型与 CRUD 打通。

交付物：
- SQLite 初始化与迁移
- `SyncTask`、`Credential`、`RepoCache`、`SyncExecution`、`SyncExecutionNode`、`WebhookEvent` 表结构
- 任务 CRUD API
- 凭证 CRUD API
- 缓存列表 API

状态：已完成

## Phase 2: 执行引擎 MVP

目标：打通“手动触发一次主仓库镜像同步”。

交付物：
- 手动触发接口
- 执行记录写入
- 本地 bare mirror 缓存创建与复用
- 主仓库镜像推送，覆盖所有分支、标签和其他 refs
- 执行状态与阶段日志
- 任务级缓存路径配置
- Git 凭证注入与日志脱敏

状态：已完成

## Phase 3: 自动建仓

目标：把目标仓库缺失时的自动创建能力接入主流程。

交付物：
- GitHub Provider
- Gogs Provider
- 目标仓库存在性检查
- 缺失仓库自动创建
- 建仓结果记录到执行节点
- 主仓库与子模块分离的 Git / API 凭证

状态：已完成

## Phase 4: 递归子模块同步

目标：支持主仓库携带多层子模块时的递归同步。

交付物：
- `.gitmodules` 解析
- 子模块引用 commit 获取
- 递归执行树
- 循环依赖检测
- 子模块目标仓库映射
- `.gitmodules` URL 改写
- 父仓库 gitlink 改写到“同步后的子模块提交”
- 树控件化执行详情视图
- 子模块独立源/目标/建仓凭证配置

状态：已完成

补充说明：
- 递归子模块模式下，目标分支会变成“可直接使用的派生提交”，因此目标分支 HEAD 不再等同于源分支 HEAD。
- 标签和其他非分支 refs 仍保持原始 mirror 语义；只有分支 refs 会覆盖为改写后的提交。

## Phase 5: 调度与 Webhook

目标：补齐非手动触发方式。

交付物：
- Cron 调度
- GitHub Webhook
- Gogs Webhook
- 任务级并发保护
- Webhook 签名校验、事件过滤、分支过滤
- Webhook 历史、重放、CSV 导出
- SSE / WebSocket 执行实时流

状态：已完成

## Phase 6: 管理台完善

目标：把后端能力完整沉淀到页面上。

交付物：
- 任务列表页
- 任务编辑页
- 执行详情页
- 缓存管理页
- 凭证页
- 调度状态面板
- Webhook 历史与状态统计
- 执行树节点详情和实时日志

状态：已完成

## 当前结论

首版 roadmap 已全部完成，当前代码库具备：
- 主仓库与递归子模块镜像同步
- 目标仓库缺失即自动建仓
- 镜像所有分支、标签和其他 refs
- 子模块 URL 与 gitlink 闭环改写
- 任务级缓存路径
- Git / API / 子模块分离凭证
- 调度、Webhook、历史重放、实时日志流
- 前端完整管理台
- 发布构建脚本、启动脚本与部署文档
